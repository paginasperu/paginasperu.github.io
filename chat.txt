Ocupa espacio en pantalla, reduciendo “visibilidad limpia” de la portada.

En móviles pequeños, puede tapar parte de la primera portada si no hay padding suficiente.

💡 Recomendación UX:

4️⃣ Cabecera que se oculta al bajar y reaparece al subir

Esto mejora UX para “modo foco total”, sin perder navegación.
Idea:

Detectar scroll vertical.

Si el usuario baja → ocultar header (transform: translateY(-100%))

Si sube → mostrar header (transform: translateY(0))

Transición suave con transition: transform 0.3s ease.

Ejemplo de JS + CSS minimalista:

header {
    transition: transform 0.3s ease;
}
header.hidden {
    transform: translateY(-100%);
}
let lastScroll = 0;
const header = document.querySelector('header');

window.addEventListener('scroll', () => {
    const currentScroll = window.pageYOffset;
    if (currentScroll > lastScroll && currentScroll > 50) {
        // bajando
        header.classList.add('hidden');
    } else {
        // subiendo
        header.classList.remove('hidden');
    }
    lastScroll = currentScroll;
});

currentScroll > 50 evita que el header desaparezca al inicio de la página.

Resultado: navegación rápida cuando se necesita, barra fuera de la vista cuando quieres foco total.

    """Sustituye marcadores de fecha en la URL original del diario."""
    f = obtener_fecha_lima()
    reemplazos = {
        "{YYYY}": f.strftime("%Y"),
        "{MM}": f.strftime("%m"),
        "{DD}": f.strftime("%d"),
        "{YY}": f.strftime("%y")
    }
    for marcador, valor in reemplazos.items():
        url_plantilla = url_plantilla.replace(marcador, valor)
    return url_plantilla

def sanitizar_nombre(diario):
    """Convierte nombres seguros para archivos."""
    if pd.isna(diario): return "desconocido"
    texto = str(diario).lower().strip()
    texto = unicodedata.normalize('NFKD', texto).encode('ascii', 'ignore').decode('ascii')
    return re.sub(r'[^a-z0-9]', '', texto)

def optimizar_imagen(contenido, ruta_destino):
    """Tratamiento Pillow para optimización JPEG progresiva."""
    try:
        img = Image.open(BytesIO(contenido))
        if img.mode != 'RGB': 
            img = img.convert('RGB')
        img.thumbnail((1200, 1600), Image.Resampling.LANCZOS)
        img.save(ruta_destino, "JPEG", quality=85, optimize=True, progressive=True)
        return True
    except Exception as e:
        print(f"      [!] Error en procesamiento de imagen: {e}")
        return False

            try:
                res = session.get(url_real, timeout=(5, 25))
                
                if res.status_code == 200:

                        print("   [!] No es una imagen válida.")

                    if len(res.content) > 10_000_000:

    except Exception as e:
        print(f"ERROR CRÍTICO DEL SISTEMA: {e}")
    finally:
        session.close()

Sí, es irónico, pero en muchísimos proyectos automatizar al inicio cuesta más que hacerlo manualmente. Esto no es una opinión: es algo muy conocido en ingeniería de software y negocios digitales.

Hay una frase famosa en tecnología:

“Don’t automate the unknown process.”
(No automatices un proceso que aún no entiendes bien.)

Te explico por qué pasa esto.

1️⃣ Automatizar tiene un costo inicial alto

Un proceso manual puede ser:

recibir foto
renombrar
subir

Tiempo:

20–30 segundos

Automatizarlo puede implicar:

programar bot
conectar APIs
manejar errores
gestionar archivos
manejar seguridad
mantener el sistema

Eso puede tomar:

10–40 horas de trabajo

Entonces el costo inicial es enorme comparado con hacerlo a mano.

2️⃣ La automatización solo gana cuando el volumen crece

Ejemplo realista.

Si recibes:

5 menús por día

Tiempo manual:

2 minutos

En un mes:

≈ 60 minutos

Pero automatizar te pudo costar 20 horas.

Entonces tardarías meses o años en recuperar ese tiempo.

3️⃣ Los procesos cambian al inicio

Cuando un proyecto empieza, el flujo cambia mucho.

Ejemplo:

Semana 1:

restaurante envía foto

Semana 3:

restaurante envía PDF

Semana 6:

restaurante envía Canva

Si automatizaste demasiado pronto, tendrás que reprogramar todo varias veces.

4️⃣ Automatizar un proceso incorrecto empeora el problema

Si el proceso no está optimizado, la automatización solo acelera el caos.

Ejemplo:

proceso confuso
↓
automatizar
↓
proceso confuso pero más rápido

Las empresas grandes primero simplifican el proceso, luego lo automatizan.

5️⃣ El costo oculto: mantenimiento

Un sistema automatizado requiere:

mantenimiento

monitoreo

corrección de errores

actualización cuando cambian APIs

Por ejemplo, si automatizas WhatsApp con un bot no oficial:

WhatsApp cambia algo

el bot deja de funcionar

Ese detalle viene de cómo las bases de datos guardan físicamente las filas en memoria y disco.

1. Las filas se guardan como bloques de datos

En motores como MySQL o PostgreSQL, cada fila se guarda más o menos así:

[id][restaurante_id][precio][nombre][descripcion]

Cada tipo de dato ocupa cierto tamaño:

Tipo	Tamaño aproximado
INT	4 bytes
BIGINT	8 bytes
BOOLEAN	1 byte
DATE	3–4 bytes
VARCHAR	variable

Entonces una fila es básicamente una secuencia de bytes.

2. Alineación de memoria (memory alignment)

Los CPUs trabajan más rápido cuando ciertos datos están alineados en múltiplos de 4 u 8 bytes.

Ejemplo simplificado:

[INT][INT][INT][VARCHAR]

Esto queda bien alineado.

Pero si pones tipos raros entre medio:

[BOOLEAN][INT][VARCHAR][INT]

a veces el motor agrega padding (bytes vacíos) para mantener la alineación.

Eso puede aumentar el tamaño de cada fila ligeramente.

3. Por eso algunas guías recomiendan

Agrupar columnas por tipo:

id (INT)
restaurante_id (INT)
precio (INT)

activo (BOOLEAN)

nombre (VARCHAR)
descripcion (TEXT)

Esto optimiza cómo se almacenan los bytes.

4. Pero en la práctica casi nadie se preocupa por esto

Porque el impacto es mínimo.

Lo que realmente afecta rendimiento es:

Índices

Cantidad de filas

Joins

Filtros (WHERE)

tamaño de datos TEXT o JSON

No el orden de columnas.

5. Dónde sí importa mucho el orden

En índices compuestos.

Ejemplo:

INDEX(restaurante_id, precio)

Esto es rápido para:

WHERE restaurante_id = 5
WHERE restaurante_id = 5 AND precio > 20

Pero no para:

WHERE precio > 20

Ahí el orden sí cambia todo.

✅ Resumen

Orden de columnas:

casi irrelevante

solo afecta alineación de bytes (muy poco)

Orden en índices:

extremadamente importante
